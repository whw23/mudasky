# 全站功能补全 + E2E 测试 + UI 美化 实施计划

## 总目标

补全所有后端已有但前端缺失的功能入口，所有内容管理统一到 web-settings 页面的所见即所得编辑中，升级编辑器，全量 E2E 测试覆盖，UI 审查美化。

---

## 核心架构要求

- **所有接口必须符合网关 RBAC 鉴权设计**（路径前缀自动推导权限）
- **内容管理全部放在 /admin/web-settings**：文章、分类、院校、案例的管理入口不再单独页面，而是在 web-settings 的对应 tab/区域中，所见即所得地编辑
- **侧边栏菜单**：根据截图，管理后台侧边栏包含：管理仪表盘、用户管理、角色管理、文章管理、分类管理、院校管理、案例管理、通用配置、网页设置。但实际内容管理（文章/分类/院校/案例）的编辑入口在 web-settings 页面的预览中，侧边栏链接跳转到 web-settings 对应区域

## Phase 1: web-settings 所见即所得编辑 ✅（侧边栏已扩展）

### 1.1 web-settings 页面重构
- 页面顶部预览区：显示公开官网的实时预览（iframe 或组件渲染）
- 预览区域上叠加编辑入口（EditableOverlay）
- 点击预览区的各个板块 → 弹出对应的编辑面板
- 可编辑区域包括：
  - **Header**：Logo、品牌名、热线电话
  - **导航栏**：菜单项
  - **Banner**：图片、标题文字
  - **首页板块**：关于我们、精选服务、热门留学国家、最新资讯、CTA
  - **院校列表**：在预览中点击 → 打开院校管理面板
  - **文章/新闻**：在预览中点击 → 打开文章管理面板
  - **成功案例**：在预览中点击 → 打开案例管理面板
  - **Footer**：公司介绍、联系方式、ICP 备案

### 1.2 图片管理
- 所有页面上"黑的地方"（Banner、Logo等）允许管理员上传替换图片
- 图片存储在后端（复用文档上传接口或新增配置图片接口）
- 上传后实时预览效果

## Phase 2: 文章编辑器升级

### 2.1 Markdown 编辑器 ✅
- `@uiw/react-md-editor`（MIT，支持源码/预览/分屏切换）
- 已替换 TiptapEditor

### 2.2 文件类型支持
- 文章类型：`markdown`（默认）、`file`（上传文件）
- 支持上传 Office 文档（.doc/.docx/.xls/.xlsx/.ppt/.pptx）和 PDF
- 上传后存储为文章附件
- PDF 预览：浏览器内嵌 `<iframe>` 或 `<embed>`
- Office 预览：使用微软 Office Online Viewer 或本地方案

### 2.3 文章详情页渲染
- Markdown 类型：`react-markdown` 渲染
- 文件类型：内嵌预览器 + 下载按钮

## Phase 3: 院校管理增强

### 3.1 省份/城市筛选
- 后端 University 模型已有 `city` 字段
- 需要后端加 `/public/university/cities` 端点（按国家筛选城市列表）

## Phase 5: 全量 E2E 测试

### 5.1 认证流程
- 手机验证码登录（正向 + 错误验证码）
- 账号密码登录（正向 + 错误密码 + 2FA 流程）
- 注册（正向 + 手机号已存在）
- 登出（验证 cookie 清除，刷新不再登录）

### 5.2 用户中心
- 个人资料编辑（用户名修改、手机号修改）
- 密码修改
- 2FA 启用/禁用
- 会话管理（踢出设备）
- 文档上传/下载/删除
- 注销账号

### 5.3 管理后台
- 用户管理：行内展开面板、状态切换、角色分配、密码重置、删除用户
- 角色管理：CRUD、权限分配
- 文章管理：CRUD、发布/取消、置顶
- 分类管理：CRUD
- 院校管理：CRUD
- 案例管理：CRUD
- 通用设置：国家码编辑
- 网站设置：品牌编辑、所见即所得编辑

### 5.4 安全/越权测试
- 未登录访问 portal/admin → 重定向
- 普通用户访问 admin → 重定向
- 未登录直接调用 admin API → 401
- XSS 注入测试（搜索框、用户名、文章内容）
- SQL 注入测试（搜索参数、路径参数）
- 接口参数篡改（无效 JSON、超长字符串、缺少必填字段）
- CSRF 防护测试

### 5.5 公开页面深度测试
- 导航栏所有链接可达
- 院校搜索/国家筛选
- 语言切换
- Footer 内容

### 5.6 全交互测试要求
- 所有按钮、链接、输入框、下拉选择、切换、弹窗都要测试
- 正例（操作成功）和反例（操作失败/输入错误）
- 接口篡改、注入攻击
